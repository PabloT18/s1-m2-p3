1. Contexto de la evaluación

En un sistema de mesa de ayuda institucional se registran tickets de soporte técnico. Cada ticket contiene información básica sobre el código del caso, el técnico asignado y el nivel de prioridad.

El objetivo de esta evaluación es implementar una clase controladora que procese una lista de tickets usando correctamente estructuras Set y Map.

En el enunciado se entregan los nombres de los métodos y sus parámetros, pero no se entrega el tipo de retorno. Se debe definirlo correctamente según la lógica solicitada.

5. Método A: filtrarYOrdenarTickets

Implementar el método filtrarYOrdenarTickets(List<TicketSoporte> tickets, int prioridadMinima).

Este método debe filtrar y ordenar tickets según la prioridad mínima indicada.

Reglas funcionales:

  • Recibe una lista de tickets de soporte.
  • Recibe un número entero que representa la prioridad mínima.
  • Solo se deben considerar tickets con prioridad mayor o igual a la prioridad mínima.
  • El resultado debe eliminar duplicados lógicos.
  • El resultado debe quedar ordenado automáticamente desde la estructura seleccionada.

Reglas de ordenamiento:

  • Ordenar por prioridad de forma descendente.
  • Si dos tickets tienen la misma prioridad, ordenar por nombre del técnico de forma ascendente.
  • La comparación del nombre del técnico debe ignorar mayúsculas y minúsculas.

Regla de unicidad:

  • Dos tickets se consideran duplicados si tienen el mismo código ignorando mayúsculas y la misma prioridad.
  • Cuando exista un duplicado lógico, solo debe conservarse un registro.

Implementación esperada:

  • Debe utilizar una estructura de tipo Set.
  • La implementación seleccionada debe permitir ordenamiento automático.
  • El ordenamiento y la unicidad lógica deben resolverse mediante un comparador.

6. Método B: agruparCodigosPorPrioridad

Implementar el método agruparCodigosPorPrioridad(List<TicketSoporte> tickets).

Este método debe agrupar tickets según el rango de prioridad de los tickets atendidos.

Rangos obligatorios:

  • ALTA: prioridad mayor o igual a 4.
  • MEDIA: prioridad igual a 2 o 3.
  • BAJA: prioridad igual a 1.

Reglas funcionales:

  • El resultado debe organizarse por clave de rango de prioridad.
  • Las claves del mapa deben ser exactamente ALTA, BAJA y MEDIA.
  • Las claves deben mantenerse ordenadas alfabéticamente.
  • El valor asociado a cada clave debe contener los NUMEROS únicos de los codigos.
  • Se deben almacenar únicamente el númerto del ticket, no objetos TicketSoporte.

Regla de códigos:

  • De cada codigo se debe tomar únicamente el numero.
  • La unicidad se determina por ese numero.
  • Si el ticket es TK-104, solo se debe almacenar el numero 104
  • Si existen numeros repetidos dentro del mismo grupo, solo se debe conservar una aparición.
  • El orden de los numeros dentro de cada grupo debe estar ordenados asendentemente

Ejemplo:

new TicketSoporte("TK-101", "Carlos Vega", 4)
new TicketSoporte("TK-102", "Carlos Mendez", 5)

En este caso, dentro del grupo ALTA se deben almacenar los números 101 y 102, porque ambos tickets tienen prioridad alta.

9. Reglas importantes

  • No modificar la clase TicketSoporte.
  • No cambiar el nombre de los métodos solicitados.
  • No cambiar los parámetros de los métodos solicitados.
  • No resolver la lógica principal dentro de Main.java.
  • No imprimir dentro de los métodos evaluados.
  • No usar Stream API.
  • No usar Collectors.
  • No usar librerías externas.
  • No usar métodos de ordenamiento de listas para reemplazar el uso de estructuras ordenadas.
  • Los métodos deben procesar datos y retornar estructuras.
